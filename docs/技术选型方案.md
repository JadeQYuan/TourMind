# 智游管家 TourMind — 技术选型方案

> 版本：v1.0  
> 日期：2026-04-16

---

## 一、选型原则

| 原则 | 说明 |
|------|------|
| **轻量优先** | 并发用户 ≤ 20，无需分布式架构，优先选择单体应用 |
| **开发效率** | 小团队快速交付，选择生态成熟、学习曲线平缓的技术 |
| **运维简单** | 尽量减少中间件依赖，一台服务器即可支撑 |
| **低成本** | 优先开源免费方案，云服务器月费 ≤ ¥200 |

---

## 二、技术栈选型

### 2.1 前端

| 技术 | 选择 | 理由 |
|------|------|------|
| 框架 | **Vue 3** | 生态完善，Composition API 开发效率高，比 React 学习曲线更平 |
| 语言 | **TypeScript** | 类型安全，减少运行时错误，IDE 友好 |
| UI 组件库 | **Ant Design Vue 4.x** | 提供完善的表单、表格、日期选择等业务所需组件，风格专业 |
| 构建工具 | **Vite** | 开发启动秒级，热更新快 |
| 状态管理 | **Pinia** | Vue 官方推荐，比 Vuex 更简洁 |
| 路由 | **Vue Router 4** | Vue 官方路由 |
| HTTP 客户端 | **Axios** | 成熟稳定，拦截器机制便于统一处理 token 和错误 |
| 图表 | **ECharts** | 财务看板折线图/柱状图 |
| PDF 生成 | **html2pdf.js** | 前端直接渲染合同 HTML → PDF，无需后端额外处理 |
| 图片上传 | **Ant Design Vue Upload** | 内置组件，支持预览、拖拽、多文件上传、进度展示 |
| 拖拽排序 | **Vue Draggable Plus** | 行程节点拖拽排序 |

### 2.2 后端

| 技术 | 选择 | 理由 |
|------|------|------|
| 框架 | **Python FastAPI** | 开发速度快，自动生成 OpenAPI 文档，异步支持好，性能优于 Flask/Django REST |
| 语言 | **Python 3.12** | 团队熟悉度高，生态丰富 |
| ORM | **SQLAlchemy 2.x + Alembic** | 成熟 ORM，Alembic 管理数据库迁移 |
| 数据验证 | **Pydantic v2** | FastAPI 内置，自动校验请求参数 |
| 认证 | **JWT (python-jose)** | 无状态认证，适合前后端分离 |
| 密码加密 | **passlib + bcrypt** | 行业标准密码哈希 |
| 文件上传 | **本地存储 / MinIO** | 小体量用本地磁盘存储，后期可迁移 MinIO |
| 文件解析 | **python-multipart** | FastAPI 处理 multipart/form-data 上传的必要依赖 |
| 任务调度 | **APScheduler** | 轻量级定时任务（如每日备份），无需 Celery |

### 2.3 数据库

| 技术 | 选择 | 理由 |
|------|------|------|
| 主数据库 | **PostgreSQL 16** | 比 MySQL 更严格的 SQL 标准，JSON 字段支持好（存行程节点），稳定可靠 |
| 缓存 | **不引入 Redis** | 用户量小，FastAPI 内存缓存 + 合理索引已够用 |

> **为什么不用 SQLite？**  
> SQLite 写操作锁粒度粗，多用户并发写入时可能出现锁等待。PostgreSQL 虽多一个服务，但部署简单，后续扩展无障碍。

### 2.4 部署 & 基础设施

| 技术 | 选择 | 理由 |
|------|------|------|
| 容器化 | **Docker + Docker Compose** | 环境一致，一条命令启动所有服务 |
| Web 服务器 | **Nginx** | 反向代理 + 静态资源托管 + HTTPS 证书 |
| WSGI 服务器 | **Uvicorn** | FastAPI 官方推荐，高性能 ASGI 服务器 |
| HTTPS | **Let's Encrypt (Certbot)** | 免费 SSL 证书，自动续签 |
| 服务器 | **阿里云/腾讯云 2核4G** | 月费约 ¥80–120，足够支撑 ≤ 20 并发 |
| 数据库备份 | **pg_dump + cron** | 每日凌晨备份到本地 + 对象存储 |

## 六、文件上传方案

### 6.1 使用场景

- 行程每日明细的相关凭证（機票/酒店订单截图等）
- 合同签署扫描件
- 账单收支凭证（转账截图、发票拍照）

### 6.2 设计约束

| 项目 | 规格 |
|------|------|
| 支持格式 | JPG、JPEG、PNG、WEBP |
| 单张大小上限 | 10 MB |
| 行程每天凭证数 | 最多 20 张 |
| 合同/账单附件数 | 最多 10 张 |

### 6.3 存储目录结构

```
uploads/
├── itineraries/
│   └── {itinerary_id}/
│       └── day_{n}/
│           └── {uuid}.{ext}
├── contracts/
│   └── {contract_id}/
│       └── {uuid}.{ext}
└── bills/
    └── {bill_id}/
        └── {uuid}.{ext}
```

- 文件名统一用 **UUID** 重命名，避免原始文件名冲突和路径遍历泈露
- 数据库存储文件**相对路径**（如 `itineraries/1/day_1/abc.jpg`），访问 URL 由服务端拼接基础路径返回

### 6.4 上传流程

```
前端选择文件
  │
  ▼
前端校验（这些检查在提交后端前进行）：格式、大小
  │
  ▼
POST /api/v1/files/upload
  ├─ multipart/form-data
  ├─ 字段：file（文件内容）、scene（itinerary/contract/bill）、related_id
  └─ 返回：{ file_url, file_key }
  │
  ▼
关联 file_key 到行程日明细 / 合同 / 账单记录
```

### 6.5 安全考虑

- 后端二次校验文件类型（通过 **MIME 类型** 和文件头魔数字，不信任文件后缀）
- 上传接口必须携带有效 JWT Token，匹名用户对该资源有写入权限
- Nginx 静态文件目录仅允许 GET 请求，不允许目录遇历
- 外部分享页面中的凭证图片 URL 含有时效签名参数（有效期 24 小时），防止永久外泄

### 6.6 前后端切分

| 层 | 工作 |
|-----|------|
| 前端 | 使用 `a-upload` 组件，`action` 指向上传接口；`accept="image/*"` 限制格式；`list-type="picture-card"` 展示缩略图 |
| 后端 | `python-multipart` + FastAPI `UploadFile`；`Pillow`（可选）用于尺寸校验和生成缩略图 |
| 存储 | 开发阶段将文件写入本地 `uploads/` 目录；链接 Docker volume 持久化；生产随时可替换为阿里云 OSS |
| 静态服务 | Nginx `location /uploads/` 直接服务文件，不经过 FastAPI |

### 4.1 后端备选：Node.js (NestJS)

| 维度 | FastAPI (Python) | NestJS (Node.js) |
|------|-----------------|-----------------|
| 开发速度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 性能 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 自动文档 | ✅ 内置 Swagger | 需安装插件 |
| 团队接受度 | Python 更通用 | 前后端同语言有优势 |
| **结论** | ✅ **推荐** | 次选 |

### 4.2 前端备选：React + Next.js

| 维度 | Vue 3 | React + Next.js |
|------|-------|-----------------|
| 学习曲线 | 平缓 | 较陡（JSX + 生态选择多） |
| 企业级 UI 库 | Ant Design Vue 完善 | Ant Design React 同等完善 |
| SSR 需求 | 无需 SSR | Next.js SSR 对本项目过重 |
| **结论** | ✅ **推荐** | 次选 |
