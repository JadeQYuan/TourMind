## 一、整体架构

系统采用**前后端分离的单体应用架构**，部署在单台云服务器上，通过 Docker Compose 编排所有服务。

```
                         ┌──────────────────────────────────────────────┐
                         │                 云服务器 (2核4G)               │
                         │                                              │
  用户浏览器               │  ┌──────────────────────────────────────────┐  │
  ──────────             │  │              Docker Compose               │  │
  Chrome/Edge  ─HTTPS──► │  │                                          │  │
                         │  │  ┌──────────┐     ┌───────────────────┐ │  │
                         │  │  │  Nginx   │────►│  前端静态文件      │ │  │
                         │  │  │ :443/:80 │     │  (Vue 3 Build)    │ │  │
                         │  │  │          │     └───────────────────┘ │  │
                         │  │  │          │                           │  │
                         │  │  │          │     ┌───────────────────┐ │  │
                         │  │  │  /api/*  │────►│   FastAPI         │ │  │
                         │  │  │          │     │   + Uvicorn       │ │  │
                         │  │  └──────────┘     │   :8000           │ │  │
                         │  │                   └────────┬──────────┘ │  │
                         │  │                            │            │  │
                         │  │                   ┌────────▼──────────┐ │  │
                         │  │                   │   PostgreSQL 16   │ │  │
                         │  │                   │   :5432           │ │  │
                         │  │                   └───────────────────┘ │  │
                         │  │                                          │  │
                         │  │  ┌──────────────────────────────────┐   │  │

                         │  │  │  - 每日 02:00 数据库备份          │   │  │
                         │  │  └──────────────────────────────────┘   │  │

                         │                                              │
```

## 三、服务拆分与通信

| 服务 | 容器名 | 端口 | 职责 |
|------|--------|------|------|
| nginx | tourmind-nginx | 80, 443 | HTTPS 终止、静态文件服务、API 反向代理 |
| backend | tourmind-api | 8000 (内网) | FastAPI 业务 API |
| db | tourmind-db | 5432 (内网) | PostgreSQL 数据持久化 |

> Nginx、FastAPI、PostgreSQL 在同一 Docker 网络内通过容器名互访，**仅 Nginx 的 80/443 端口对外暴露**。

## 四、API 设计规范

### 4.1 URL 规范

```
/api/v1/{模块}/{资源}
```

### 4.2 认证方式

- 登录接口 `POST /api/v1/auth/login` 返回 `access_token`（有效期 8h）
- 所有业务接口需在 Header 携带：`Authorization: Bearer <token>`

### 4.3 主要 API 端点

#### 认证
```
POST   /api/v1/auth/login          # 登录
POST   /api/v1/auth/refresh        # 刷新 token
GET    /api/v1/auth/me             # 获取当前用户信息
```

#### 行程管理
```
GET    /api/v1/itineraries         # 行程列表（分页+筛选）
POST   /api/v1/itineraries         # 创建行程
GET    /api/v1/itineraries/{id}    # 行程详情（含每日节点）
PUT    /api/v1/itineraries/{id}    # 更新行程
DELETE /api/v1/itineraries/{id}    # 删除行程
POST   /api/v1/itineraries/{id}/copy  # 复制行程
```

#### 合同管理
```
GET    /api/v1/contracts           # 合同列表
POST   /api/v1/contracts           # 创建合同
GET    /api/v1/contracts/{id}      # 合同详情（含行程明细+财务汇总）
PUT    /api/v1/contracts/{id}      # 更新合同
PATCH  /api/v1/contracts/{id}/status  # 变更合同状态
GET    /api/v1/contracts/{id}/pdf  # 导出合同 PDF（后端生成备用）
```

#### 账单管理
```
GET    /api/v1/bills               # 账单列表
POST   /api/v1/bills               # 创建账单（入账/出账）
GET    /api/v1/bills/{id}          # 账单详情
PUT    /api/v1/bills/{id}          # 更新账单
DELETE /api/v1/bills/{id}          # 删除账单
GET    /api/v1/bills/summary       # 财务汇总（Dashboard 数据）
GET    /api/v1/bills/export        # 导出账单 Excel
```

#### 文件上传
```
POST   /api/v1/files/upload        # 上传凭证/附件，返回 file_url
```

### 4.4 统一响应格式

```json
{
  "code": 0,
  "message": "success",
  "data": { ... }
}
```

错误响应：
```json
{
  "code": 40001,
  "message": "合同不存在",
  "data": null
}
```

## 五、数据流说明

### 5.1 创建合同的数据流

```
前端选择行程
    │
    ▼
GET /api/v1/itineraries/{id}
    │  返回行程完整数据（含每日节点 JSON）
    ▼
前端展示行程明细，用户可在合同内微调
    │
    ▼
POST /api/v1/contracts
    {
      itinerary_id: 原行程 ID（仅记录来源）,
      itinerary_snapshot: { ... }  ← 行程明细快照，独立存储
      customer_info: { ... },
      pricing: { ... }
    }
    │
    ▼
数据库：contracts 表 + contract_itinerary_days 表
```

> **快照设计**：合同创建时将行程明细复制一份存储（快照），与原行程解耦。后续修改原行程模板不影响已签合同内容。

### 5.2 账单与合同关联

```
收入账单 ──关联──► 合同
出账账单 ──关联──► 合同（可选）

合同详情页
  聚合查询：
    已收金额 = SUM(bills WHERE contract_id=? AND type='income')
    总支出   = SUM(bills WHERE contract_id=? AND type='expense')
    待收     = contract.total_amount - 已收金额
```

## 六、安全设计

| 威胁 | 防护措施 |
|------|---------|
| 未授权访问 | JWT 认证，所有 API 必须携带有效 token |
| 暴力破解密码 | 登录接口限流：同 IP 5 次失败后锁定 30 分钟 |
| SQL 注入 | 使用 SQLAlchemy ORM，禁止拼接原始 SQL |
| XSS | 前端 Vue 模板自动转义，后端不返回原始 HTML |
| CSRF | 前后端分离 + JWT，无 Cookie 认证，天然防 CSRF |
| 敏感数据泄露 | 密码使用 bcrypt 哈希，日志不打印敏感字段 |
| 文件上传风险 | 限制文件类型（jpg/png/pdf）和大小（≤10MB），存储路径随机化 |
| HTTPS | Nginx + Let's Encrypt，强制 HTTPS 跳转 |

## 七、部署方案

### 7.1 Docker Compose 服务配置概览

```yaml
# docker-compose.yml（示意）
services:
  nginx:
    build: ./nginx
    ports: ["80:80", "443:443"]
    volumes:
      - ./frontend/dist:/usr/share/nginx/html  # 前端静态文件
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf
      - certbot-certs:/etc/letsencrypt
    depends_on: [backend]

  backend:
    build: ./backend
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/tourmind
      - SECRET_KEY=${SECRET_KEY}
    volumes:
      - uploads:/app/uploads
    depends_on: [db]

  db:
    image: postgres:16-alpine
    environment:
      - POSTGRES_DB=tourmind
      - POSTGRES_USER=user
      - POSTGRES_PASSWORD=${DB_PASSWORD}
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
  uploads:
  certbot-certs:
```

### 7.2 部署步骤

```bash
# 1. 服务器初始化
sudo apt update && sudo apt install docker.io docker-compose-plugin -y

# 2. 克隆代码
git clone <repo> && cd tourmind

# 3. 配置环境变量
cp .env.example .env
vi .env   # 填写数据库密码、JWT Secret、域名等

# 4. 构建前端
cd frontend && npm install && npm run build

# 5. 启动所有服务
docker compose up -d

# 6. 首次运行数据库迁移
docker compose exec backend alembic upgrade head

# 7. 申请 SSL 证书
docker compose run --rm certbot certonly --webroot ...
```

### 7.3 日常运维

```bash
# 查看日志
docker compose logs -f backend

# 手动备份数据库
docker compose exec db pg_dump -U user tourmind > backup_$(date +%Y%m%d).sql

# 更新部署
git pull && docker compose up -d --build backend
```

## 八、监控与告警（轻量方案）

| 监控项 | 方案 |
|--------|------|
| 服务存活 | Docker `restart: unless-stopped` 自动重启 |
| 磁盘用量 | 云服务器控制台告警（磁盘 > 80%） |
| 数据库备份 | cron 每日执行，备份文件上传 OSS，同时发送微信通知 |
| 错误日志 | FastAPI 日志写文件，定期人工检查或配置简单告警 |

## 九、扩展性说明

当前单体架构满足 ≤ 20 并发用户，若未来业务增长，可按以下路径扩展：

```
阶段 1（当前）: 单机 Docker Compose
    │
    ▼
阶段 2（百人级）: 升级服务器配置（4 核 8G）+ PostgreSQL 读写分离
    │
    ▼
阶段 3（千人级）: Kubernetes 编排 + 微服务拆分 + CDN 加速
```

目前的模块划分（行程/合同/账单）已按业务边界隔离，便于未来微服务化改造。
