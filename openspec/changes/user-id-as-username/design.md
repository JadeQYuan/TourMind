## Context

当前系统所有涉及“用户名”的功能，均调整为统一使用“工号”字段（六位数字，自动递增，唯一且不可编辑），不新增新模块，仅在现有 user 相关代码内调整。

## Goals / Non-Goals

**Goals:**
- 全系统统一用工号（六位数字，自动递增，唯一且不可编辑）替代用户名
- 工号在用户列表中展示，并支持查询
- 工号可用于登录
- 工号创建后不可修改

**Non-Goals:**
- 不涉及历史用户数据的批量迁移
- 不支持自定义工号格式

## Decisions

- 所有“用户名”相关字段、接口、表单、展示、校验等，均统一替换为“工号”
- 工号字段在数据库中设置为唯一且不可更改（仅新建用户时分配）
- 前端用户表单中，工号字段只读，新增时自动生成，编辑时不可修改
- 登录接口仅支持通过工号+密码登录
- 用户列表支持按工号查询

## Risks / Trade-offs

- [风险] 历史数据中存在非六位数字工号 → [缓解] 暂不处理历史数据，仅保证新建和后续操作符合新规则
- [风险] 工号分配逻辑出错导致重复 → [缓解] 后端分配时加唯一约束并回滚冲突
- [风险] 工号字段只读后，特殊场景下无法修改 → [缓解] 如需修改需走特殊流程，由管理员介入

## Migration Plan

1. 后端创建用户时自动分配下一个可用工号，保证唯一性（仅新建用户）
2. 前端用户表单调整，工号字段只读且自动生成，所有涉及用户名的地方统一替换为工号
3. 登录接口仅支持工号登录
4. 用户列表支持按工号查询
5. 测试所有相关功能，确保无遗漏

## Open Questions

- 历史用户数据如有非六位数字工号，是否需要后续补齐？
- 工号分配是否允许跳号（如删除用户后工号是否复用）？建议不复用
